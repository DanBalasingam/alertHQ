AlertHQ

Design direction

Industrial dark. Operator-first.

AlertHQ is a tool used under pressure. The aesthetic should feel like a control room — precise, dense, fast to scan. No gradients, no playfulness. Information hierarchy is everything.

Visual principles

Information density
Operators need to see a lot at once. Compact rows, small labels, meaningful colour only.
Severity-first colour
Colour encodes severity, not branding. Red means critical, always. Never use red for anything else.
Monospace data
Timestamps, IDs, metric names, host labels — always monospace. Aligns columns, looks authoritative.
No motion for motion's sake
Animation only when it conveys state change — e.g. a firing alert pulsing. Never decorative.

Tone references

PagerDuty — information density, dark theme · Grafana — monospace data, severity colour · Linear — compact rows, keyboard-first · Datadog — timeline, alert feed density

What AlertHQ isn't: Notion (too soft), Jira (too noisy), Slack (too conversational).

Page-level guidance

Per-page design notes

Specific layout and component decisions for each route.

/dashboard

Stat strip: 4 numbers — Firing, Acknowledged, Open Incidents, Resolved (24h). Use coloured values, not coloured backgrounds.
Two-column below: Left = recent alert feed (compact rows). Right = open incidents list.
No charts in v1 — Grafana handles that. Keep dashboard as a status summary, not a metrics dashboard.

/alerts

Table layout — not cards. Alerts are dense data, tables scale better.
Columns: Severity bar (3px) · Alert name · Source · Labels/tags · Status · Linked incident · Age. All sortable.
Filter bar: Status chips (All / Firing / Pending / Resolved) + search. Persistent across sessions via URL params.
Row click: Slide-over panel (not full page nav) for alert detail + "Create incident" CTA.

/incidents & /incidents/:id

List: Card layout (not table) — incidents have more metadata than alerts. Left border = severity.
Detail: Two-column — left (flex-1) = incident info + linked alerts + description. Right (320px fixed) = timeline + actions.
Actions: Acknowledge, Resolve, Link alert, Add comment — always visible in right panel, not buried in menus.
Status transitions: Open → Acknowledged → Resolved. Show as a simple 3-step stepper at the top of the detail page.

/incidents/new — create from template

Step 1: Select template (card grid, searchable). Selected template highlighted with accent border.
Step 2: Pre-filled form from template — title, severity, description, runbook URL. All editable.
Step 3: Link alerts (multi-select from firing alerts list).
Keep it single-page — don't use a wizard with separate route steps. Just scroll sections with a sticky step indicator.

/templates & /templates/:id/edit

List: Card grid (2-col). Card shows name, description, severity default, last used.
Editor: Split — left = form fields (name, severity, description, runbook URL, custom fields). Right = live preview of what the incident will look like when created.
Custom fields: Operators can add key-value pairs to the template. Drag to reorder. Shown in incident detail.

Slide-over panel — reusable pattern

Use for alert detail, quick incident view, template preview. Slides in from the right at 480px wide.
Background dims to rgba(0,0,0,0.5) but stays interactive — user can still see the list behind it.
Close on Escape or clicking the backdrop. Never use a full-page route for content that's a subset of the list view.
